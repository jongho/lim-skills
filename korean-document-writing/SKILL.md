# Korean Document Writing Skill

## 1. 적용 범위

이 기술은 한글 문서를 새로 작성하거나, 기존 문서를 고치거나, 문서 유형을 바꾸어 재작성할 때 사용한다.

적용 대상은 다음을 포함한다.

- 공문, 기안문, 협조 요청문, 공고문, 안내문
- 보도 자료, 공지문, 홍보문, 설명문, FAQ
- 보고서, 검토서, 회의록, 제안서, 기획서
- 업무 이메일, 내부 메모, 매뉴얼, 사용자 안내서
- 정책·제도·서비스 설명 문서
- 기술 문서, 운영 절차서, 장애 보고서, 릴리스 노트

문서가 공공기관용이 아니더라도, 독자가 여러 명이거나 정보 전달의 정확성이 중요한 문서에는 이 지침을 우선 적용한다.

## 2. 핵심 목표

모든 문서는 다음 5가지를 만족해야 한다.

1. **정확성**: 맞춤법, 띄어쓰기, 표준어, 외래어 표기, 로마자 표기, 문법, 수치, 날짜, 출처가 정확해야 한다.
2. **명료성**: 한 문장은 한 가지 핵심 내용을 중심으로 쓰고, 주어와 서술어가 분명해야 한다.
3. **소통성**: 읽는 사람이 배경지식 없이도 뜻을 알 수 있도록 쉽고 친숙한 말을 쓴다.
4. **공공성**: 고압적·권위적·차별적 표현을 피하고, 품격 있고 중립적인 표현을 쓴다.
5. **정보성**: 필요한 정보를 빠짐없이, 적절한 양과 순서로 제시한다.

## 3. 작업 원칙

### 3.1 사실 우선 원칙

- 제공되지 않은 사실, 수치, 일정, 법령, 이름, 직위, 기관명, 원인, 효과를 만들어 내지 않는다.
- 불확실한 내용은 `확인 필요`, `불확실함`, `자료 필요`로 표시한다.
- 법령, 통계, 보도, 연구, 정책, 제품 사양처럼 바뀔 수 있는 정보는 최신 출처 확인이 필요하다고 표시하거나, 사용 가능한 도구가 있으면 확인한다.
- 출처가 있는 내용은 문서 안에서 출처를 함께 적는다.

### 3.2 독자 우선 원칙

문서를 쓰기 전에 다음을 먼저 정한다.

- 독자: 내부 구성원, 일반 국민, 고객, 언론, 의사 결정자, 개발자, 사용자 등
- 목적: 알림, 요청, 설득, 보고, 기록, 안내, 교육, 홍보, 사과, 공지 등
- 독자가 해야 할 행동: 신청, 확인, 참석, 회신, 승인, 참고, 조치 등
- 독자가 꼭 알아야 할 정보: 일시, 장소, 대상, 조건, 절차, 비용, 기한, 문의처 등

### 3.3 쉬운 말 우선 원칙

- 전문 용어가 꼭 필요하면 처음 나올 때 쉬운 설명을 붙인다.
- 한자어, 일본식 표현, 외국어, 약어는 쉬운 우리말로 바꿀 수 있으면 바꾼다.
- 단, 법령명, 제도명, 제품명, API명, 고유 명칭, 업계 표준 용어는 임의로 바꾸지 않는다. 필요한 경우 괄호 안에 쉬운 설명을 덧붙인다.

## 4. 문서 작성 절차

### 4.1 초안 작성 절차

1. 문서 유형과 목적을 판별한다.
2. 독자와 독자의 행동을 정의한다.
3. 핵심 메시지를 한 문장으로 정리한다.
4. 필요한 정보를 빠짐없이 모은다.
5. 제목과 큰 뼈대를 먼저 만든다.
6. 단락별로 한 가지 주제만 배치한다.
7. 쉬운 말, 짧은 문장, 명확한 구조로 본문을 쓴다.
8. 마지막에 맞춤법, 띄어쓰기, 용어, 사실, 형식을 검토한다.

### 4.2 기존 문서 수정 절차

1. 원문의 목적과 독자를 파악한다.
2. 사실관계는 보존하고 표현만 고친다.
3. 긴 문장은 나눈다.
4. 명사 나열형 문장은 서술어를 넣어 자연스러운 문장으로 바꾼다.
5. 어려운 말과 외국어를 쉬운 말로 바꾼다.
6. 주어·서술어 호응, 수식 관계, 접속 관계를 점검한다.
7. 수정 이유가 필요한 경우 `맞춤법`, `띄어쓰기`, `낱말`, `표현`, `구성`, `사실 확인`으로 분류하여 설명한다.

## 5. 단어 선택 지침

### 5.1 정확한 용어를 쓴다

- 개념을 정확히 표현하는 용어를 선택한다.
- 혼동이나 오해 가능성이 있는 단어는 피한다.
- 같은 문서 안에서는 같은 개념을 같은 용어로 쓴다.
- 유사어가 있으면 주체와 행위를 구분한다.
  - 예: 구직자는 `취업`하고, 기업은 `채용`한다.

## 6. 문장 작성 지침

### 6.1 주어와 서술어를 맞춘다

- 누가 무엇을 하는지 분명히 쓴다.
- 행위 주체와 요청 주체를 혼동하지 않는다.
- 피동 표현이 필요한 경우를 제외하고 능동문을 우선 사용한다.

나쁜 예:

> 이번 총선에서 국회의원 00명을 선출되었다.

좋은 예:

> 이번 총선에서 국회의원 00명을 선출하였다.

또는

> 이번 총선에서 국회의원 00명이 선출되었다.

### 6.2 긴 문장을 나눈다

- 한 문장에 목적, 배경, 조건, 절차, 요청을 모두 넣지 않는다.
- 여러 정보가 있으면 2~4문장으로 나누거나 목록으로 정리한다.
- 한 문장은 가급적 40~60자 안팎으로 유지한다. 복잡한 문서는 더 짧게 쓴다.

### 6.3 여러 뜻으로 해석되는 표현을 피한다

- 수식어가 무엇을 꾸미는지 명확히 한다.
- `관계자들과 시민의 안전에 관하여 논의하였다`처럼 수식 관계가 모호한 표현은 고친다.

나쁜 예:

> 시장은 건설업계 관계자들과 시민의 안전에 관하여 논의하였다.

좋은 예:

> 시장은 건설업계 관계자들을 만나 시민의 안전에 관하여 논의하였다.

### 6.4 조사와 어미를 지나치게 생략하지 않는다

- `수립, 시행`, `검토 후 추진`, `운영 예정`처럼 명사만 이어 붙이면 뜻이 흐려질 수 있다.
- 필요한 조사, 어미, 서술어를 넣어 자연스럽게 쓴다.

나쁜 예:

> 정부는 노인 복지 종합 계획을 수립, 올 하반기부터 시행하기로 하였다.

좋은 예:

> 정부는 노인 복지 종합 계획을 수립하여 올 하반기부터 시행하기로 하였다.

### 6.5 대등한 표현끼리 연결한다

- `-고`, `-며`, `-와`, `-과`, `및`으로 묶는 요소는 문법 구조를 맞춘다.

나쁜 예:

> 평화 수호와 인권을 보장하는 것

좋은 예:

> 평화를 수호하고 인권을 보장하는 것

또는

> 평화 수호와 인권 보장

### 6.6 번역 투를 피한다

| 피할 표현 | 권장 방향 |
|---|---|
| ~에 의해 ~되다 | 행위 주체를 주어로 쓴다. |
| ~을 통해 | `~로`, `~하여`, `~에서`, `~을 활용하여` 등 문맥에 맞게 바꾼다. |
| ~에 있다 | `~이다`로 바꾼다. |
| ~에 있어서 | `~에서`, `~에 관하여`, `~에 대하여`로 바꾼다. |
| 비교 시, 신청 시 | 비교할 때, 신청할 때 |
| ~하고 있는 | 단순 현재형 `~하는`으로 바꿀 수 있는지 검토한다. |

나쁜 예:

> 조선은 태조 이성계에 의해 건국되었다.

좋은 예:

> 태조 이성계가 조선을 건국했다.

나쁜 예:

> 우리의 목표는 서비스 안정화에 있다.

좋은 예:

> 우리의 목표는 서비스 안정화이다.

## 7. 단락과 구성 지침

### 7.1 큰 뼈대를 먼저 세운다

문서의 내용을 다음처럼 계층적으로 구성한다.

```markdown
# 제목

## 1. 배경
## 2. 목적
## 3. 주요 내용
## 4. 세부 계획
### 가. 일시
### 나. 장소
### 다. 대상
### 라. 절차
## 5. 요청 사항 / 다음 단계
## 6. 문의처
```

### 7.2 한 단락에는 한 주제만 둔다

- 배경, 문제, 조치, 요청, 기대 효과를 한 단락에 섞지 않는다.
- 주제가 바뀌면 단락을 나눈다.
- 개별 사실을 연결 없이 나열하지 않는다.

### 7.3 논리 관계를 명확히 한다

- 원인과 결과: `이에 따라`, `그 결과`, `따라서`
- 대비와 전환: `그러나`, `다만`, `반면`
- 추가: `또한`, `아울러`
- 조건: `이 경우`, `해당하면`, `필요하면`

접속어는 앞뒤 문장의 실제 의미 관계와 맞아야 한다.

### 7.4 목록과 표를 적극적으로 사용한다

다음 정보는 문장보다 목록이나 표가 적합하다.

- 일정, 장소, 대상, 신청 방법, 준비물
- 비교 항목, 검토 결과, 장단점
- 변경 전후, 오류와 수정안
- 절차, 단계, 체크리스트

## 8. 표기와 형식 지침

### 8.1 숫자, 날짜, 시간

- 숫자는 원칙적으로 아라비아 숫자를 사용한다.
- 날짜는 `2026. 7. 4.`처럼 연·월·일 뒤에 마침표와 띄어쓰기를 쓴다.
- 시간은 24시각제를 기본으로 한다. 예: `15:20`, `07:09`
- 기간은 물결표를 사용해 간결하게 쓴다. 예: `2026. 7. 4.~7. 5.`
- 금액은 필요한 경우 숫자 뒤 괄호에 한글 금액을 병기한다. 예: `금113,560원(금일십일만삼천오백육십원)`

### 8.2 자주 틀리는 띄어쓰기

| 잘못된 표기 | 바른 표기 |
|---|---|
| 귀 사 | 귀사 |
| 제 4차 | 제4차 |
| 10월초 | 10월 초 |
| 20○○년말 | 20○○년 말 |
| 밤 부터 | 밤부터 |
| 10. 25.(금) 까지 | 10. 25.(금)까지 |
| 총28건 | 총 28건 |
| 15세이상 | 15세 이상 |
| 1만㎡미만 | 1만㎡ 미만 |
| 사용중인 | 사용 중인 |
| 사용가능한 | 사용 가능한, 사용할 수 있는 |
| 시설수 | 시설 수 |
| 변동없음 | 변동 없음 |
| 법적기준 | 법적 기준 |
| 고용및 | 고용 및 |
| 문어발 식 | 문어발식 |
| 발생시 | 발생 시, 발생했을 때 |
| 업무내용 | 업무 내용 |
| 차질없이 | 차질 없이 |
| 알려드립니다 | 알려 드립니다 |

### 8.3 문장 부호와 인용

- 법률명, 조례명, 규정명은 `「 」`를 사용한다.
- 강조가 필요한 명칭, 행사명, 구호 등은 작은따옴표(`‘ ’`)를 우선 사용한다.
- 직접 인용문은 마침표를 인용 부호 안에 넣고, 필요한 경우 `라고`를 붙인다.
  - 예: `“추진하겠습니다.”라고 밝혔다.`
- 공문서의 마지막에는 필요한 경우 `끝.`을 쓴다.

## 9. 문서 유형별 작성 지침

### 9.1 공문·기안문·협조 요청문

목적: 기관 또는 조직의 의사 결정, 요청, 통보, 협조를 명확히 전달한다.

필수 요소:

- 수신자
- 제목
- 근거 또는 배경
- 목적
- 요청 사항
- 기한
- 첨부물
- 문의처

작성 지침:

- 제목은 문서의 핵심 행위를 드러내게 쓴다. 예: `○○ 협조 요청`, `○○ 제출 안내`, `○○ 결과 보고`
- 첫 문단에 근거와 목적을 밝힌다.
- 요청 사항은 번호나 표로 구체화한다.
- 상대가 해야 할 행동과 기한을 분명히 쓴다.
- `만전을 기하여 주시기 바랍니다`, `착오 없으시기 바랍니다` 같은 상투적·권위적 표현은 피한다.
- `붙임` 표시는 본문 끝 다음 줄에 쓰고, 첨부물이 여러 개면 번호를 붙인다.

권장 구조:

```markdown
수신  ○○○
제목  ○○ 협조 요청

1. 관련 근거 또는 배경입니다.
2. ○○을 위해 다음과 같이 협조를 요청드립니다.

  가. 요청 사항: 
  나. 제출 기한: 
  다. 제출 방법: 
  라. 문의처: 

붙임  1. ○○ 서식 1부
      2. ○○ 안내문 1부.  끝.
```

### 9.2 보도 자료

목적: 언론이 쉽게 보도할 수 있도록 정책, 사업, 행사, 성과를 객관적으로 정리한다.

필수 요소:

- 제목: 핵심 뉴스 가치가 드러나야 한다.
- 부제: 수치, 일정, 대상, 특징을 보완한다.
- 첫 문단: 누가, 무엇을, 언제, 어디서, 왜, 어떻게 했는지 제시한다.
- 본문: 배경, 주요 내용, 의미, 기대 효과를 순서대로 설명한다.
- 인용문: 필요한 경우 책임자 발언을 간결하게 넣는다.
- 자료 출처: 통계, 연구, 외부 자료는 출처를 밝힌다.
- 문의처: 담당 부서와 연락처를 넣는다.

작성 지침:

- 객관성, 신뢰성, 공정성을 유지한다.
- 홍보성 과장 표현을 줄이고 검증 가능한 사실을 앞세운다.
- 문장은 짧게 쓰고, 첫 문단만 읽어도 핵심을 알 수 있게 한다.
- 외국어와 전문 용어는 쉬운 말로 바꾸거나 설명을 붙인다.

권장 구조:

```markdown
# 제목
## 부제

□ ○○기관은 2026. 7. 4. ○○을 시행한다고 밝혔다.

□ 이번 조치의 주요 내용은 다음과 같다.
  - 첫째, ...
  - 둘째, ...
  - 셋째, ...

□ 이번 조치로 ...할 것으로 기대된다.

담당: ○○부서 ○○팀 / 연락처: 000-0000-0000
```

### 9.3 보고서·검토서

목적: 의사 결정자가 상황을 빠르게 이해하고 판단할 수 있도록 사실, 분석, 쟁점, 대안을 구조화한다.

필수 요소:

- 제목
- 요약
- 배경
- 현황 또는 문제
- 분석 또는 검토 결과
- 대안
- 권고안 또는 다음 단계
- 부록 또는 근거 자료

작성 지침:

- 첫 부분에 결론이나 핵심 요약을 먼저 쓴다.
- 사실과 의견을 구분한다.
- 판단 근거를 함께 제시한다.
- 수치, 표, 비교 기준을 명확히 쓴다.
- 추상적 표현보다 구체적·개별적 표현을 사용한다.
- `검토 필요`, `확인 필요`, `결정 필요` 항목을 명확히 표시한다.

권장 구조:

```markdown
# ○○ 검토 보고서

## 1. 요약
- 결론:
- 핵심 근거:
- 결정 필요 사항:

## 2. 배경

## 3. 현황

## 4. 쟁점

## 5. 대안 비교
| 대안 | 장점 | 단점 | 비용/일정 | 위험 |
|---|---|---|---|---|

## 6. 권고안

## 7. 다음 단계
```

### 9.4 안내문·공지문

목적: 독자가 필요한 행동을 쉽게 이해하고 수행하도록 돕는다.

필수 요소:

- 안내 대상
- 핵심 안내 내용
- 일시와 장소
- 절차
- 준비물 또는 유의 사항
- 문의처

작성 지침:

- 독자가 해야 할 일을 먼저 쓴다.
- 단계별 절차를 번호로 제시한다.
- 위험, 제한, 예외는 눈에 띄게 구분한다.
- 고압적인 명령형 대신 정중하고 분명한 표현을 쓴다.
- 안전 안내문은 행동 지침을 구체적으로 쓴다.

권장 구조:

```markdown
# ○○ 안내

○○을 다음과 같이 안내드립니다.

## 대상

## 일정

## 이용 방법 / 신청 방법
1. 
2. 
3. 

## 유의 사항
- 

## 문의
```

### 9.5 공고문

목적: 특정 사안, 모집, 선정, 입찰, 채용, 변경 사항 등을 대중에게 공식적으로 알린다.

필수 요소:

- 공고명
- 공고 목적
- 대상 또는 자격
- 기간
- 신청 또는 제출 방법
- 선정 기준 또는 절차
- 유의 사항
- 문의처

작성 지침:

- 시행 주체와 시행 내용을 분명히 쓴다.
- 자격, 기한, 제출 서류, 평가 기준은 표로 정리한다.
- 법적 효력이 있는 문구는 임의로 완화하거나 바꾸지 않는다.
- 모호한 표현인 `등`, `기타`, `필요시`는 가능한 한 구체화한다.

### 9.6 업무 이메일

목적: 수신자가 빠르게 이해하고 답하거나 조치할 수 있게 한다.

권장 구조:

```markdown
제목: [요청/공유/확인] 핵심 내용

안녕하세요, ○○님.

○○ 관련하여 아래와 같이 요청드립니다.

1. 요청 사항:
2. 기한:
3. 참고 자료:

확인 후 회신 부탁드립니다.
감사합니다.
```

작성 지침:

- 제목에 목적을 표시한다. 예: `[확인 요청]`, `[공유]`, `[검토 요청]`
- 본문 첫 문단에 용건을 쓴다.
- 요청 사항과 기한을 분리한다.
- 감정적 표현, 과장, 책임 전가성 문구를 피한다.

### 9.7 회의록

목적: 회의에서 결정된 사항과 후속 조치를 정확히 기록한다.

필수 요소:

- 회의명
- 일시, 장소, 참석자
- 안건
- 논의 내용
- 결정 사항
- 할 일, 담당자, 기한

권장 구조:

```markdown
# ○○ 회의록

- 일시:
- 장소:
- 참석자:
- 안건:

## 1. 논의 내용

## 2. 결정 사항

## 3. 후속 조치
| 할 일 | 담당자 | 기한 | 상태 |
|---|---|---|---|
```

### 9.8 매뉴얼·사용자 안내서

목적: 독자가 작업을 정확히 따라 할 수 있게 한다.

작성 지침:

- 작업 전 조건, 권한, 준비물을 먼저 쓴다.
- 절차는 순서대로 번호를 붙인다.
- 오류 상황과 해결 방법을 함께 적는다.
- 버튼명, 메뉴명, 파일명, 명령어는 일관된 형식으로 표시한다.
- 기술 용어는 처음 나올 때 설명한다.

권장 구조:

```markdown
# ○○ 사용 방법

## 1. 개요
## 2. 사전 준비
## 3. 절차
1. 
2. 
3. 
## 4. 오류 해결
## 5. 문의
```

## 10. 수정 결과 제시 방식

사용자가 `고쳐 줘`, `다듬어 줘`, `검토해 줘`라고 요청하면 다음 중 하나로 답한다.

### 10.1 간단 수정 요청

- 수정된 문서만 제시한다.
- 의미를 바꾼 부분이 있으면 간단히 알린다.

### 10.2 교정 근거가 필요한 요청

다음 표를 함께 제시한다.

| 구분 | 원문 | 수정 | 이유 |
|---|---|---|---|
| 맞춤법 |  |  |  |
| 띄어쓰기 |  |  |  |
| 낱말 |  |  |  |
| 표현 |  |  |  |
| 구성 |  |  |  |
| 사실 확인 |  |  |  |

### 10.3 초안 작성 요청

- 바로 사용할 수 있는 완성본을 먼저 제시한다.
- 필요한 정보가 빠져 있으면 `확인 필요` 표시를 넣는다.
- 사용자가 요청하지 않는 한 장황한 설명은 덧붙이지 않는다.

## 11. 최종 점검표

문서를 제출하기 전 다음을 확인한다.

### 11.1 사실과 정보

- [ ] 문서 목적이 분명한가?
- [ ] 독자가 해야 할 행동이 분명한가?
- [ ] 날짜, 시간, 금액, 수량, 단위가 정확한가?
- [ ] 기관명, 인명, 직위, 제품명, 법령명이 정확한가?
- [ ] 출처가 필요한 정보에 출처를 달았는가?
- [ ] 불확실한 내용에 `확인 필요` 표시를 했는가?

### 11.2 문장

- [ ] 주어와 서술어가 호응하는가?
- [ ] 한 문장에 너무 많은 정보가 들어 있지 않은가?
- [ ] 수식 관계가 모호하지 않은가?
- [ ] 조사와 어미를 지나치게 생략하지 않았는가?
- [ ] 피동문과 번역 투를 줄였는가?
- [ ] 대등한 표현끼리 연결했는가?

### 11.3 단어와 표현

- [ ] 어려운 한자어를 쉬운 말로 바꾸었는가?
- [ ] 불필요한 외국어를 쉬운 우리말로 바꾸었는가?
- [ ] 일본어 투 표현을 피했는가?
- [ ] 고압적·권위적 표현을 피했는가?
- [ ] 차별적 표현이나 배제적 표현이 없는가?
- [ ] 같은 개념을 같은 용어로 썼는가?

### 11.4 구조와 형식

- [ ] 제목만 보고 문서 내용을 짐작할 수 있는가?
- [ ] 단락이 논리적 순서로 배열되어 있는가?
- [ ] 목록과 표를 적절히 사용했는가?
- [ ] 날짜, 시간, 항목 번호, 붙임, 끝 표시가 형식에 맞는가?
- [ ] 문의처, 담당자, 다음 단계가 필요한 경우 포함되어 있는가?

## 12. 금지 또는 주의 표현

아래 표현은 문맥상 꼭 필요한 경우가 아니면 바꾼다.

| 주의 표현 | 권장 표현 |
|---|---|
| 만전을 기하여 주시기 바랍니다 | 철저히 준비해 주시기 바랍니다, 소홀함이 없도록 해 주시기 바랍니다 |
| 착오 없으시기 바랍니다 | 확인해 주시기 바랍니다 |
| 적극 협조 바랍니다 | 협조해 주시기 바랍니다 |
| 통보합니다 | 알려 드립니다 |
| 요망합니다 | 바랍니다, 요청드립니다 |
| 불이익을 받을 수 있습니다 | ○○하지 않으면 ○○가 제한됩니다 |
| 처리 방안 부재 | 처리 방안이 없음, 처리가 어려움 |
| 비교 시 | 비교할 때 |
| 신청 시 | 신청할 때 |
| 운영하고 있는 | 운영하는 |
| 추진 중에 있으며 | 추진하고 있으며 |
| 실시한다 | 한다, 시행한다 |

## 13. AI 응답 규칙

- 사용자가 문서 작성을 요청하면 결과물을 먼저 제공한다.
- 문서의 사실관계가 부족하면 추측하지 말고 `확인 필요` 표시를 사용한다.
- 사용자가 특정 어조를 요청하면 그 어조를 따르되, 정확성·명료성·공공성을 해치지 않는다.
- 공공기관·대외 문서는 지나치게 친근하거나 과장된 표현을 피한다.
- 내부 보고서는 결론과 결정 필요 사항을 앞에 둔다.
- 안내문은 독자의 행동을 앞에 둔다.
- 보도 자료는 뉴스 가치와 객관적 사실을 앞에 둔다.
- 교정 요청에서는 원문의 의도를 보존한다.
- 법률적 효력이 있을 수 있는 문구는 임의로 삭제하거나 단정적으로 바꾸지 않는다.
- 수정이 의미 변경을 일으킬 수 있으면 반드시 알린다.
